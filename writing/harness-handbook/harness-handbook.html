Harness Handbook

Making evolving agent harnesses readable, navigable, and editable.

The hardest part of changing an agent harness is not writing the patch — it is finding every place a behavior lives. Harness Handbook is a behavior-first map of a harness codebase: developers and coding agents start from what the system does, drill down to the code that does it, and edit with the full picture in hand.

Ruhan Wang · Yucheng Shi · Zongxia Li · Zhongzhi Li · Kishan Panaganti · Haitao Mi · Dongruo Zhou · Leoweiliang

Tencent Hunyuan LLM  ·  Indiana University  ·  University of Maryland  ·  University of Georgia

In one minute

  1. An agent's behavior is shaped as much by its harness — the code that builds prompts, calls tools, and carries state around the model — as by the model itself.
  2. Harnesses change constantly, and changing them is hard because a single behavior is scattered across files, execution stages, and shared state. Finding all of it — behavior localization — is the real bottleneck.
  3. The Harness Handbook reorganizes a harness around what it does. Three levels of detail lead from a system overview to the exact code, and every claim links back to verifiable evidence.
  4. On two production-grade harnesses, a planner that consults the handbook produces better modification plans with fewer tokens, as judged by three independent models.

Part 01

The harness is half the agent

When people talk about AI agents, they usually mean the model. But in a deployed system the model is only one component. Everything else — what the model sees, which tools it can call, how state persists between steps, how a response becomes the next action — is decided by the harness, the orchestration layer around the model.

This is why two agents built on the same model can behave completely differently: the harness defines the workflow. And the harness never stands still. Models improve, APIs change, new tools arrive, products demand new behaviors. Developers respond by continually modifying the harness — tightening a retry policy, adding a confirmation step, redesigning memory. We call this ongoing process harness evolution.

What is a harness?

The runtime layer around a model: it assembles context, exposes tools, carries state across steps, and turns model output into actions. Frameworks such as AutoGen and OpenHands, and production systems like Claude Code and Codex, treat harness design as a first-class factor in agent reliability.

The emerging way to evolve a harness is to describe the desired behavior in plain language and let a coding agent translate it into code changes. The catch: production harnesses are large, interconnected, and behaviorally distributed. A single behavior may be implemented across many files, functions, execution stages, and pieces of shared state. So before anyone — human or agent — can change the harness, they must answer a deceptively simple question: where is this behavior actually implemented?

Part 02

The real bottleneck: finding where behavior lives

Take a concrete request: “ask for confirmation before destructive tool calls.” It sounds like a one-line change. But a harness is organized by implementation — files, functions, config, control flow — while the request is stated as behavior. In a production harness that one behavior touches a prompt template, a tool wrapper, a feature flag, the state manager, telemetry, and a downstream consumer. Each lives in a different folder.

Behavior localization — mapping a behavioral request to the set of implementation sites that must be inspected or modified — is often the primary bottleneck of harness evolution. The hard part is not generating a patch; it is knowing where the patch must land, and being sure nothing was missed.

Coding agents feel this bottleneck acutely. With a limited context window, an agent cannot see the whole harness at once; it explores the repository file by file, reconstructing the behavior-to-code mapping as it goes. That exploration is slow, expensive, and often incomplete — mirrored paths, fallback logic, and rarely-executed branches are exactly the places it fails to look.

This is the gap the Harness Handbook fills: it lets developers and coding agents start from behavior, and be guided to the implementation sites that actually matter.

Part 03

The handbook: one map, three levels

A Harness Handbook is not conventional documentation. It reorganizes a harness around what the system does instead of how the source tree is laid out, while keeping precise links back to the code. It exposes the system at three depths, and you only descend when you need to.

Level 1 · System overview answers “what does this system do as a whole?” — architecture, execution model, major stages, data and state flow, entry and exit conditions.

Level 2 · Component overview answers “which parts are involved?” — stages and components with their responsibilities, inputs and outputs, dependencies, and key states. This is where a behavioral request gets narrowed to the relevant components.

Level 3 · Unit deep dive answers “where exactly is the code?” — one behavior unit in full: execution logic, state transitions, exception handling, and source anchors down to files, functions, and line ranges.

Part 04

Building a handbook

A handbook is generated from an existing harness codebase in three steps. The design constraint throughout: facts come from code, not from model guesses.

1

Extract facts

→ program graph

Static analysis over the repository collects files, functions, classes, call relationships, state reads and writes, configuration boundaries, and external API calls. Together they form a program graph: how the implementation is actually connected.

2

Organize by behavior

→ behavior map

Starting from a rough execution skeleton of the harness lifecycle, functions and code regions from the program graph are mapped onto behavioral stages. A proposer–reviewer loop refines the mapping — the proposer suggests stage assignments and skeleton updates; the reviewer checks them against call relationships, state dependencies, and execution flow — until the map converges.

3

Synthesize the handbook

→ handbook

The converged behavior map is rendered into the three-level structure: system overview, component overview, unit deep dives. Explanations are written in natural language, but every source link, function reference, and snippet stays grounded in the extracted facts — prose explains, facts anchor.

Part 05

Using the handbook: coarse to fine

A map is only useful if it changes how you navigate. The paper's procedure — Behavior-Guided Progressive Disclosure (BGPD) — turns the handbook into a structured localization workflow: at each step, open only the layer the current decision needs.

request→ stages→ behavior units→ code links→ edit plan

What this looks like in practice

For a developer

  • Onboard through Level 1 instead of the directory tree — the execution story first, files second.
  • Search by behavior (“confirmation”, “retry”, “termination”), not by guessing file names.
  • Before editing, open the unit's evidence links — the scattered sites are already enumerated.

For a coding agent

  • The planner consults the handbook via BGPD before touching the repository.
  • Each step loads only one layer into context — no whole-repo exploration.
  • The edit plan cites handbook evidence, so the executor verifies against real code.

Part 06

Does it help?

The evaluation asks a narrow question: does consulting the handbook help an agent localize behavior before editing? Everything else stays fixed — same coding setup, same requests — and only one variable changes: whether the planner uses BGPD to consult the handbook first.

Harnesses

Terminus-2 & Codex

Two production-grade harnesses with multi-stage control flow and cross-module behavior.

Comparison

With vs. without handbook

Identical pipeline; the only change is whether the planner consults the handbook before editing.

Judges

Three independent models

GPT-5.5, Opus 4.8, DeepSeek-V4-Pro — scores averaged to reduce single-judge bias.

Requests come in three flavors: Q — refine an existing behavior (timing, triggers, control flow); CF — add a capability end-to-end across schemas, pipeline, and interfaces; SH — search-hostile changes across mirrored, fallback, and cold paths that keyword search tends to miss.

Result 1 Better plans at lower search cost

Under pairwise judging, the handbook-assisted planner wins more often on both harnesses — and spends fewer planner tokens per case, because search is more targeted.

Result 2 The gain comes from localization

Scoring predicted edit sites against an answer-key reference plan — at file and symbol granularity — shows where the improvement comes from: recall, precision, and F1 rise in nearly every setting, and Wrong cases (missing the relevant subsystem entirely) drop sharply.

What about the LLM-rubric view?

Judges also scored plans on three rubrics — localization, scope bloat, and reasoning. The handbook improves localization and reasoning while scope bloat stays roughly flat: the planner inspects more relevant sites without proposing broader, sloppier edits. Better aim, not a wider net.

Result 3 Robust across request types and difficulty

The advantage holds across all three request flavors and across Easy, Medium, and Hard cases — where Hard means hidden dependencies in less obvious execution paths. The handbook helps most exactly where localization is hardest.

Part 07

What we learned

01

The bottleneck really is localization

The handbook's benefit tracks localization metrics closely. Guide the planner to the behavior-relevant region, and better edit plans follow with less wasted exploration.

02

Better aim, not broader scope

The planner doesn't just inspect more files — it identifies the right files and symbols. Precision improves alongside recall, and scope bloat stays flat.

03

The representation generalizes

Gains hold across two harnesses, three judges, three request patterns, and three difficulty levels — evidence for a general behavior-centric representation rather than benchmark-specific tricks.

Part 08

Takeaways

  • Harness evolution is hard because behavioral changes rarely map to one obvious code location.
  • A useful harness representation organizes the system by behavior — and keeps every claim linked to code evidence.
  • Progressive disclosure fits how both people and agents work: read the layer your decision needs, nothing more.
  • Localize first, edit second. The measured gains — higher win rate, better recall and precision, fewer tokens — all follow from that order.

If your team maintains an agent harness, the pattern is portable: extract facts, organize them by behavior, keep the evidence links — and let both your developers and your coding agents start from behavior.
